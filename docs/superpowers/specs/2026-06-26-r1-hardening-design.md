# R1 Hardening Design

## Goal

Harden the completed local MVP without changing the core LLM Wiki model.

R1 keeps the Karpathy LLM Wiki loop intact: raw sources stay preserved, the Wiki remains the durable Markdown artifact, and schema/lint rules keep the maintained knowledge layer healthy. It improves reliability around CLI automation, broken-vault detection, invalid raw-source paths, and documentation.

## Scope

R1 includes four areas:

1. CLI machine-readable output.
2. Broken-vault and frontmatter lint hardening.
3. Clearer `.raw/` ingest path errors.
4. R1 documentation and progress records.

## CLI Output

All commands keep the current human-readable text output by default.

Each command also accepts `--json`. In JSON mode:

- successful commands print one JSON object to stdout;
- failed commands print one JSON object to stdout and return exit code `1`;
- error objects include `operation`, `status`, `error.type`, and `error.message`;
- dataclass results are serialized through their public fields.

This gives Codex adapters, scripts, and future CI checks a stable interface without breaking current CLI usage.

## Frontmatter Lint

The current lint only checks whether a page starts with a YAML frontmatter delimiter. R1 upgrades this to validate required fields:

- `type`
- `title`
- `created`
- `updated`
- `status`

Missing fields are reported as high-severity `frontmatter-field` findings. Malformed frontmatter delimiters are reported as high-severity `frontmatter` findings.

R1 intentionally does not add a full YAML parser dependency. The MVP frontmatter generated by core is simple line-oriented metadata, so a conservative line parser is enough for this stage.

## Raw Source Path Errors

`ingest` should fail with actionable messages when the user supplies an invalid source path:

- absolute path outside `.raw/`;
- relative path outside `.raw/`;
- traversal path such as `.raw/../outside.md`;
- missing raw file.

Core raises clear exceptions. CLI text mode prints a concise error. CLI JSON mode returns structured error data.

## Fixtures And Documentation

R1 adds fixture coverage for stricter lint behavior and updates docs:

- `docs/roadmap-schedule.md` marks R1 as complete after implementation.
- `docs/release-notes-v0.1.0-mvp.md` records R1 as post-MVP hardening on `main`.
- `codex_doc/project_understanding_progress.md` records implementation evidence outside the package repository.

## Non-Goals

- Do not implement Obsidian CLI actual read/write/search.
- Do not implement URL ingest.
- Do not implement batch ingest.
- Do not implement deep retrieval or vector search.
- Do not change the existing Markdown artifact layout.
- Do not move the `v0.1.0-mvp` tag.
