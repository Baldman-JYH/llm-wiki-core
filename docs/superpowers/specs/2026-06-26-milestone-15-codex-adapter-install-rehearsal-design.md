# Milestone 15 Codex Adapter Install Rehearsal Design

## Goal

Verify that the repo-local Codex adapter installer creates a vault that is immediately usable from Codex App / CLI through discoverable `AGENTS.md` instructions and the installed `llm-wiki` command.

## Context

Milestone 14 proved the installed core CLI can run the MVP Wiki loop. Milestone 15 verifies the Codex adapter path around that core:

- installer entrypoint;
- generated vault instructions;
- natural language command discovery;
- re-entry through `status` and `continue`;
- lint-clean artifact output.

The adapter must serve the LLM Wiki workflow without becoming a second behavior source. Core operations remain the authority for artifacts.

## Scope

Implement and verify:

- `AGENTS.md` generated by `llm-wiki init` contains the MVP Codex command list.
- `AGENTS.md` generated by `llm-wiki init` contains natural language trigger mapping.
- `integrations/codex/COMMANDS.md` remains a valid Markdown command table.
- `integrations/codex/install/install.ps1` can run against a temporary vault.
- The installer-created vault can run `status`, `continue`, and `lint`.

## Non-Goals

- Do not install global Codex skills.
- Do not modify user-level Codex configuration.
- Do not publish a marketplace plugin.
- Do not implement slash-command registration.
- Do not implement actual Obsidian CLI read/write/search calls.
- Do not modify `D:/ai/llmWiki/claude-obsidian`.

## Design Decision

The generated `AGENTS.md` should include Codex command discovery directly, because the repo-local installer currently creates vaults through `llm-wiki init`. This keeps the first-run user experience coherent without requiring global Codex configuration changes.

The generated instructions remain thin: they point Codex to neutral `llm-wiki` commands instead of duplicating operation behavior in adapter prose. This preserves the neutral-core design while making the local Codex user path practical.
